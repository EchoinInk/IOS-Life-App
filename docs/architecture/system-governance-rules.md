# System Governance Rules

This document defines the guardrails that protect the platform's philosophy. This is the governance layer that prevents emotional, cognitive, and adaptive drift.

---

## Purpose

Governance ensures:
- Every feature aligns with the adaptive philosophy
- No feature increases shame, overwhelm, or cognitive load
- The system remains emotionally safe and low-energy friendly
- The platform stays an adaptive behavioral system, not a productivity tool

---

## Core Governance Principles

- **The system must protect user emotional safety** — No feature may cause shame, anxiety, or emotional harm.
- **Cognitive load must never increase without justification** — Every addition must be offset by simplification elsewhere.
- **Low-energy compatibility is mandatory** — All features must function when the user has minimal capacity.
- **Features must adapt to user state** — Rigid, one-size-fits-all features are prohibited.
- **No rigid workflows** — Users must never be forced into predetermined paths.
- **No punitive mechanics** — Nothing in the system punishes inaction, inconsistency, or disengagement.
- **No moral framing** — Features must never imply moral judgment about user behavior.
- **No shame-based nudges** — All prompts, reminders, and notifications must use shame-free language.
- **Every feature must have a clean exit** — Users can abandon any flow at any point without penalty or residue.

---

## Cognitive Load Governance

### Required Evaluation
Every feature must be evaluated for cognitive impact:
- What cognitive load does this add?
- What cognitive load does this remove?
- Does this introduce new objects, screens, or modes?
- Does this require memory or multi-step reasoning?
- Does this create branching paths?
- Does this increase density?

### The Rule
**If cognitive load increases without compensating reduction, the feature is rejected.**

---

## Low-Energy Mode Governance

### Required Evaluation
Every feature must be evaluated for low-energy compatibility:
- How does this behave during burnout?
- Can this be done in under 10 seconds?
- Can this be done with one thumb?
- Does this punish low-energy states?
- Does this require consistency the user cannot maintain?

### The Rule
**If a feature only works when the user is at full capacity, it cannot ship.**

---

## Emotional Safety Governance

### Required Evaluation
Every feature must be evaluated for emotional impact:
- What emotional tone does this create?
- Does this introduce shame, pressure, or urgency?
- Could this overwhelm the user?
- Does this create a sense of falling behind?
- Does this feel punitive?

### The Rule
**If a feature risks emotional harm, it must be redesigned.**

---

## Adaptive Compatibility Governance

### Required Evaluation
Every feature must be evaluated for adaptability:
- What state does this support?
- Does it adapt to different states?
- Does it break when the user's state changes?
- Does it introduce rigid workflows?

### The Rule
**If a feature cannot adapt, it cannot ship.**

---

## Flow Exit Governance

### Required Evaluation
Every feature must be evaluated for exit safety:
- How does the user exit this flow?
- Can they exit at any time without penalty?
- Does exiting leave cognitive residue?
- Does the feature trap the user in a mode?

### The Rule
**If a user cannot exit cleanly, the feature is rejected.**

---

## Architectural Quality Control

### Mandatory Review Layers
Every feature must pass through:
1. **Design review** — Visual and interaction design evaluation
2. **Feature review checklist** — Alignment with functional requirements
3. **Emotional safety checklist** — Validation against emotional harm criteria
4. **Cognitive load checklist** — Measurement of cognitive impact
5. **Adaptive compatibility checklist** — Verification of state adaptability

### The Rule
**No feature merges without passing all five.**

---

## Philosophy Enforcement

Every feature must reinforce the platform's identity as an adaptive behavioral system.

### Identity Principles
- **Adaptive, not prescriptive** — The system changes for the user, not the reverse
- **Supportive, not demanding** — The app offers help, never commands
- **Low-energy, not high-effort** — Designed for minimal capacity states
- **Emotionally safe, not shaming** — No guilt, pressure, or judgment
- **Cognitively light, not complex** — Reduced mental effort required
- **Flexible, not rigid** — No forced workflows or mandatory paths
- **State-aware, not state-agnostic** — Recognizes and responds to user capacity

### The Rule
**If a feature violates the platform's identity, it cannot ship.**

---

*These governance rules ensure the platform remains true to its core philosophy: an adaptive, emotionally safe behavioral system that works for humans in all states of energy and capacity.*
